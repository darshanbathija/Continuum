## Vision

### 10x Check
Today's Continuum is a read-only gauge for AI agent rate limits. The 10x version makes Continuum **the universal control plane for AI coding agents across all your devices and machines.** Sessions live anywhere in your tailnet, surface in one view on your phone, get approved from your wrist, are aware of rate-limit budgets, cost-track per-session, and detect their own completion. The gauge becomes a conductor.

### Platonic Ideal
You're on the subway. Phone buzzes. Watch shows "1 plan ready." Glance — it's the Defx V3 auth refactor you started before leaving the house. Tap the complication. Plan card opens: 3 files to change, diff inline, each one expandable. Tap "Approve & run." Watch buzzes again 40 minutes later: "Done. PR #847 opened." You never opened a terminal.

When you DO open a terminal — Mac, at the desk — all 4 active sessions in a sidebar grouped by repo. One is waiting on a plan. Two are running. One is paused because Claude weekly is at 87% and would cap before completion. You toggle the rate-limit gate. Cost: $0.43 today. You feel in command.

## Approach (D1)

**Approach C — Hybrid: terminal + structured cards in v1**

tmux as PTY layer with `tmux -CC` control mode parsed by an in-process SwiftNIO daemon inside `ClawdmeterMac`. SwiftTerm renders the terminal pane. JSONL tail of `~/.claude/projects/<encoded>/<id>.jsonl` and `~/.codex/sessions/...` feeds a parallel structured-card view (messages, tool calls, plan-with-diff). Mobile defaults to structured view; Mac defaults to terminal. Toggle between them per session.

Rejected: Approach A (terminal-only, structured polish in Phase 6 — ships a worse Termius for mobile v1). Approach B (structured-only, no terminal — fails the user's explicit "view, interact, control terminals" requirement).

## Accepted Scope (added to this plan)
- **Foundation (D1)**: Hybrid terminal + structured-cards UI, both shipped in v1. Important: tmux `-CC` mode emits structured frames (`%output <octal-escaped bytes>`); the daemon parses frames and forwards the raw escaped bytes — already in xterm/ANSI form from the inner `claude`/`codex` child — directly to SwiftTerm. There is no double-rendering: tmux doesn't render in `-CC` mode, it only frames. SwiftTerm consumes the pane bytes; the daemon never tries to render tmux's own UI.
- **Notifications (D3, REVISED per D15)**: NO APNS. Instead: **(a)** when the iOS app is foregrounded, the daemon streams session events to it over the existing WebSocket; iOS posts a `UNUserNotificationCenter` local notification for plan-ready / session-done / paused events; **(b)** when the iOS app is backgrounded, iOS uses `BGAppRefreshTask` (Background App Refresh framework) to poll the daemon's `/sessions/needs-attention` endpoint every ~15–30 min (iOS-scheduled, can't be tighter); on wake, post local notifications for any pending events. **Trade-off accepted via D15**: lose the "phone buzzes within seconds of plan-ready" responsiveness; gain zero APNS key-custody surface (Codex outside-voice flagged `.p8` in DMG as a security advisory waiting to happen). **Watch complication (D10)** still works — it gets its data over WCSession from the iPhone whenever the phone wakes; the lag inherits from the iPhone's background-refresh cadence.
- **Pin-a-goal + done-detection (D4)**: Goal field on new-session sheet. JSONL detector fires only at an **end-of-turn boundary** (last event is an assistant text message and no `tool_use` is pending a `tool_result`); on every end-of-turn event we evaluate three signals: **(a) High-confidence text signal** — the final assistant message contains user-supplied goal text (lowercased substring + trim punctuation) **AND** the message contains a completion verb (`done`, `finished`, `complete`, `shipped`, `passes`, `passing`). OR **(b) Side-effect signal** — a `Bash` tool call earlier in this same turn ran `git commit` and exited 0. OR **(c) Tiebreaker** — the final assistant message contains the goal text (no completion verb required) **AND** no new JSONL events for ≥ 10 minutes after the boundary (true idle, not just slow). The guards prevent firing mid-run, during long bash/thinking blocks, OR during slow back-and-forth conversations. Session card flips to "done"; D3 push fires.
- **Worktree-on-create (D7)**: Checkbox on new-session sheet → `git worktree add .claude/worktrees/<slug> <base-branch>` where `<slug>` is `<goal-slug>-<shortid>` (e.g. `auth-tests-7f3a2c`) — goal slug derived from the goal field (lowercased, non-alphanumeric → `-`, truncated to 24 chars); shortid is the first 6 chars of the session UUID. Collisions: if the directory already exists, append `-2`, `-3`, etc. Cleanup on session delete (`git worktree remove`). Orphan GC: on app launch, scan `.claude/worktrees/*` and remove any whose session-id is no longer in the registry, after a 24h grace period (so a re-attach within a day doesn't lose the worktree).
- **Watch complication (D10)**: New `plan-waiting` complication. **v1 ships ONE family — `.accessoryCircular`** — sufficient for the platonic-ideal "glance at watch face → tap → approve" flow. The remaining three families (`.accessoryCorner`, `.accessoryRectangular`, `.accessoryInline`) ship post-v1. Tap → watchOS approval screen with plan summary + Approve button. Re-opens Watch scope after the original "skip Watch" v1 cut; one family is the minimum viable surface.

## Reviewer Concerns (open, captured for implementation)

Codex outside-voice findings folded into the implementation plan but worth re-checking during implementation:

1. **`tmux send-keys -l <bytes>` is not a byte-safe transport** for IME, paste-bursts, complex escape sequences. Phase 3 should evaluate `tmux load-buffer` + `paste-buffer` (writes to tmux's paste buffer then pastes into the target pane) as an alternative for non-trivial input. Test with: a 4KB paste, an emoji, a Ctrl-C, a vi `:wq`.
2. **JSONL tailing fragility**: `DispatchSourceVnode` alone misses rename-after-rotation and partial-line writes. Phase 2 implementation should also handle: re-opening on `EVFILT_VNODE` rename events, line-buffered reads (don't parse partial JSON), explicit handling of Claude's session-file naming pattern.
3. **Phase 0 probe needs broader coverage** than "round-trip one keystroke." Add: resize, UTF-8 boundary at chunk edges, alternate-screen entry/exit, 1MB output burst, pane death mid-stream. Document the pass criteria.
4. **Done-detection signal (c) "goal text + 10min idle" remains a false-positive factory** in conversational sessions where the user is typing slowly. Implementation should consider an explicit user-marked "done" verb-button as a higher-confidence alternative, with (c) demoted to a soft suggestion.
5. **Bind 0.0.0.0 + accept-filter remains a security smell** to Codex even with token auth. Implementation should re-evaluate during Phase 1 — if a bind-to-explicit-interface approach is feasible despite Tailscale `utun` flapping (e.g., re-bind on interface change events from `NWPathMonitor`), prefer that path. If not, the accept-filter is acceptable for personal-use scope.
6. **Token rotation/revocation UI** for pairing tokens — D3 originally had `/devices/register` with bearer-token auth; with D15 mooting that endpoint, the broader "what if I lose my iPhone" story still needs a Settings UI to **regenerate-pairing-token** and **revoke-device-token**. Add to Phase 5 (iOS pairing flow).
