# Cross-implementation test vectors — clawdmeter-relay

Per the E1 design doc (`docs/design/secure-relay-apns-2026-05-26.md` §8), the relay's wire protocol crosses TWO independent crypto stacks:

- **TypeScript / libsodium-wrappers-sumo** — runs inside the Cloudflare Worker (this folder)
- **Swift / CryptoKit Curve25519 + ChaCha20Poly1305** — runs in the Mac daemon + iOS app (lands in E3/E4)

Both stacks MUST produce byte-identical outputs for the same inputs, or paired peers will silently fail to decrypt each other's frames. These JSON fixtures are the contract.

Each vector is a small, self-contained JSON file with:

- A short `name` + `description` documenting WHAT it tests
- Fully specified inputs (deterministic — no `random()` calls allowed)
- Fully specified expected outputs (byte-exact)

The TypeScript test suite consumes these fixtures via `test-vectors.test.ts`. The Swift test suite consumes them via `apple/ClawdmeterShared/Tests/.../RelayCryptoTests.swift` (lands in E3 — at that point the fixtures here become the gating contract).

## Format notes

- All `bytes` fields are **lowercase hex**.
- The `description` field is human-readable; it doesn't gate any assertion.
- `kind` enumerates which crypto operation the vector exercises so test runners can dispatch.

## Source-of-truth + regeneration

These vectors were generated by the TypeScript test harness (`test-vectors.test.ts`) using `libsodium-wrappers-sumo`. If a vector ever needs regeneration (e.g., adding a new test case), run the generation script — never hand-edit the `expected_*` fields.

Adding a NEW vector? Append a row to the table above AND add an entry to the dispatch switch in `test-vectors.test.ts`.

## Why ChaCha20-Poly1305, not "sealed-box"?

The original PR task brief mentioned "libsodium sealed-box" envelopes. The E1 design doc (`docs/design/secure-relay-apns-2026-05-26.md` §4.2 + §4.3) specifies the more efficient flow: **peers derive a shared symmetric key via X25519 + HKDF first**, then encrypt every subsequent frame with **XChaCha20-Poly1305 AEAD** on that derived key.

From the relay's perspective the difference is immaterial — both shapes are opaque bytes the relay forwards blind. The design doc's flow:

1. Avoids the per-frame asymmetric crypto cost of sealed-box (sealed-box re-derives an ephemeral keypair per encryption)
2. Lets us reuse the same derived key for the APNS gateway envelope (saves a second ECDH)
3. Matches Apple CryptoKit's high-perf path (`ChaChaPoly.seal`)

The design doc is the source of truth because the Mac/iOS clients (E3/E4) implement against it; the relay only ever sees the opaque bytes either way.
